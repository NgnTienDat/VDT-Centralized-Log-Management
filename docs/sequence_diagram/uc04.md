### 3.3.2.4. Use case Phát hiện bất thường dựa trên quy tắc

| Thuộc tính | Nội dung |
|------------|-----------|
| **Use case ID** | UC_04 |
| **Tên use case** | Phát hiện bất thường dựa trên quy tắc |
| **Mô tả** | Theo chu kỳ đánh giá của từng quy tắc, hệ thống tự động thực thi chuỗi `pipelineSteps` đã cấu hình để truy vấn log, tính toán biểu thức và so sánh điều kiện trigger. Khi phát hiện vi phạm, hệ thống chuyển trạng thái quy tắc sang **FIRING**, ghi nhận lịch sử cảnh báo và phát thông báo qua WebSocket. |
| **Tác nhân** | Hệ thống, thông qua tiến trình định kỳ ngầm của bộ quản lý lập lịch (Scheduler Manager). |
| **Tiền điều kiện** | - Quy tắc cảnh báo có `isActive = true`.<br>- Quy tắc đã được nạp vào bộ lập lịch của hệ thống.<br>- Có `triggerStepId` hợp lệ. |
| **Hậu điều kiện** | - Trạng thái quy tắc được cập nhật.<br>- `lastRunTime` được ghi nhận.<br>- Lịch sử cảnh báo được lưu nếu phát sinh cảnh báo.<br>- Thông báo được đẩy qua WebSocket nếu điều kiện vi phạm thỏa mãn. |

#### Luồng cơ bản

| Hành động của tác nhân | Phản hồi của hệ thống |
|------------------------|-----------------------|
| **1.** Đến chu kỳ lập lịch, Scheduler Manager tự động kích hoạt tiến trình kiểm tra quy tắc. | |
| | **2.** Hệ thống lấy `pipelineSteps` của quy tắc và bắt đầu thực thi pipeline. |
| | **3.** Engine thực thi tuần tự từng bước trong pipeline. Nếu bước đầu tiên là `FETCH_ES_DATA`, hệ thống gửi truy vấn đến Elasticsearch, áp dụng bộ lọc thời gian theo `lookBackMinutes` và `@timestamp`, sau đó xử lý gom nhóm nếu có. Kết quả được lưu trong context nội bộ để các bước tiếp theo sử dụng. |
| | **4.** Nếu pipeline có bước `MATH`, hệ thống lấy dữ liệu từ context, tính toán biểu thức SpEL theo từng nhóm hoặc theo giá trị tổng quát, rồi ghi kết quả mới trở lại context. |
| | **5.** Hệ thống thực hiện bước `THRESHOLD`, duyệt qua toàn bộ nhóm dữ liệu và so sánh giá trị thực tế với ngưỡng cấu hình (`value`). Nếu phát hiện nhóm vi phạm điều kiện, hệ thống đánh dấu `triggered = true` và lưu tên nhóm vào danh sách `breachedGroups`. |
| | **6.** Nếu `triggered = true`, hệ thống kiểm tra trạng thái hiện tại của quy tắc. Nếu quy tắc chưa ở trạng thái **FIRING**, hệ thống chuyển trạng thái sang **FIRING** và cho phép gửi thông báo ngay. Nếu quy tắc đã ở trạng thái **FIRING**, hệ thống chỉ gửi thông báo khi đã hết thời gian cooldown hoặc xuất hiện nhóm vi phạm mới. |
| | **7.** Hệ thống lưu dữ liệu vi phạm vào lịch sử cảnh báo trên Elasticsearch và đồng thời publish thông báo qua WebSocket để Dashboard hiển thị ngay lập tức. |

#### Luồng thay thế

| Mã | Mô tả |
|-----|--------|
| **A1** | Nếu quy tắc không có bước nào hoặc bước đầu tiên không phải `FETCH_ES_DATA`, hệ thống dừng thực thi và ghi log lỗi. |
| **A2** | Nếu `triggerStepId` không tồn tại trong pipeline hoặc không hợp lệ, hệ thống dừng thực thi và ghi log lỗi. |
| **A3** | Nếu bước `MATH` thiếu dữ liệu đầu vào hoặc thiếu biểu thức (`expression`), hệ thống ghi lỗi kiểm tra dữ liệu và dừng pipeline. |
| **A4** | Nếu bước `EVALUATE_THRESHOLD` thiếu `inputStepId` hoặc giá trị ngưỡng, hệ thống ghi lỗi kiểm tra dữ liệu và dừng pipeline. |
| **A5** | Nếu quy tắc đang ở trạng thái **FIRING**, chưa hết thời gian cooldown và không có nhóm vi phạm mới, hệ thống không gửi thông báo, chỉ ghi nhận trạng thái hiện tại. |
| **A6** | Nếu quy tắc đã **FIRING** nhưng phát hiện nhóm vi phạm mới trong lần chạy hiện tại, hệ thống bỏ qua cooldown và gửi thông báo ngay. |

#### Luồng ngoại lệ

| Mã | Mô tả |
|-----|--------|
| **E1** | Nếu truy vấn Elasticsearch thất bại, hệ thống ghi log lỗi và không tạo cảnh báo cho lần chạy đó. |
| **E2** | Nếu `MathExecutor` hoặc `EvaluateThresholdExecutor` không thể xử lý dữ liệu đầu vào, hệ thống ghi log lỗi và kết thúc lần thực thi hiện tại. |
| **E3** | Nếu hệ thống không tìm thấy dữ liệu của bước được chọn làm trigger, pipeline bị dừng và không sinh cảnh báo hợp lệ. |

== Bước 1: Kích hoạt kiểm tra định kỳ ==

Scheduler -> Engine: Thực thi Rule\ntheo chu kỳ đánh giá

activate Engine

== Bước 2: Thực thi Pipeline ==

Engine -> ES: Truy vấn log theo pipelineSteps\n(FETCH_ES_DATA)

ES --> Engine: Trả về dữ liệu log và kết quả tổng hợp

Engine -> Engine: Thực hiện các bước tính toán\n(MATH, THRESHOLD)

== Bước 3: Đánh giá điều kiện cảnh báo ==

alt Điều kiện trigger không thỏa mãn

    Engine -> Engine: Cập nhật trạng thái Rule = OK

else Điều kiện trigger thỏa mãn

    Engine -> Engine: Kiểm tra trạng thái FIRING\nvà thời gian cooldown

    alt Được phép gửi cảnh báo

        Engine -> ES: Lưu lịch sử cảnh báo

        ES --> Engine: Xác nhận

        Engine -> WS: Publish cảnh báo

        WS -> UI: Hiển thị cảnh báo\nthời gian thực

    else Đang trong thời gian cooldown

        Engine -> Engine: Bỏ qua gửi thông báo

    end

end

== Bước 4: Cập nhật trạng thái thực thi ==

Engine -> ES: Cập nhật thông tin Rule\n(lastRunTime, alertState,...)

deactivate Engine
