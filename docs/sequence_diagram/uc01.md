
### 3.3.2.1. Use case Thu thập và xử lý – chuẩn hóa log

| Thuộc tính | Nội dung |
|------------|-----------|
| **Use case ID** | UC_01 |
| **Tên use case** | Thu thập và xử lý - chuẩn hóa log |
| **Mô tả** | Hệ thống tự động thu thập log thô từ các node ứng dụng, chuyển tiếp qua Logstash để tách trường, chuẩn hóa dữ liệu và lưu trữ tập trung vào Elasticsearch, đồng thời chuyển tiếp log đã chuẩn hóa về Backend để phát tán thời gian thực tới Frontend qua WebSocket. |
| **Tác nhân** | Filebeat Agent, Logstash, Backend hệ thống |
| **Tiền điều kiện** | - Filebeat Agent đã được cài đặt và cấu hình đúng đường dẫn tệp log trên các node ứng dụng.<br>- Logstash, Elasticsearch và Backend đang hoạt động ổn định.<br>- Các node ứng dụng đang phát sinh log ra tệp cục bộ. |
| **Hậu điều kiện** | - Dữ liệu log được chuẩn hóa thành cấu trúc có trường rõ ràng.<br>- Log được lưu thành công vào Elasticsearch.<br>- Log được chuyển tiếp sang Backend và phát ra WebSocket để Frontend nhận theo thời gian thực. |

#### Luồng cơ bản

| Hành động của tác nhân | Phản hồi của hệ thống |
|------------------------|-----------------------|
| **1.** Các dịch vụ ứng dụng ghi log ra tệp log cục bộ. | |
| **2.** Filebeat Agent giám sát file log liên tục, thu thập log mới và gắn metadata môi trường như environment, app, service. | |
| **3.** Filebeat chuyển tiếp log thô sang Logstash qua giao thức Beats trên cổng 5044. | |
| |**4.** Logstash tiếp nhận luồng log đầu vào. |
| |**5.** Logstash bóc tách các trường như log_timestamp, thread, trace_id, level, logger, raw_message,... sau đó chuẩn hóa tách main message và stack trace nếu có, đồng thời loại bỏ các trường trung gian không cần thiết. |
| | **6.** Logstash ghi dữ liệu đã chuẩn hóa vào Elasticsearch theo index pattern `sys-logs-{environment}-{date}`. |
|| **7.** Logstash gửi đồng thời log đã xử lý sang Backend qua HTTP POST tới endpoint `/internal/logs/ingest` dưới dạng JSON. |
|| **8.** Backend nhận log, phát sinh sự kiện nội bộ rồi WebSocket Publisher đẩy log ra Frontend hiển thị theo thời gian thực. |

#### Luồng thay thế

| Mã | Mô tả |
|-----|--------|
| **A1** | Log có định dạng không khớp grok pattern. Logstash gắn tag `_grokparsefailure` và bỏ qua bước ghi Elasticsearch cũng như bước gửi sang Backend. |
| **A2** | Elasticsearch không sẵn sàng hoặc lỗi ghi dữ liệu. Logstash không lưu được log vào chỉ mục đích và ghi nhận lỗi xử lý. |
| **A3** | Backend không phản hồi hoặc lỗi khi nhận webhook. Logstash không chuyển được log sang Backend; Frontend không nhận được log thời gian thực đối với log đó. |

#### Luồng ngoại lệ

| Mã | Mô tả |
|-----|--------|
| **E1** | Nếu Logstash phát hiện lỗi parsing, log thô bị loại khỏi luồng chuẩn hóa. |
| **E2** | Nếu lỗi xảy ra trong lúc Backend xử lý thu thập log, hệ thống ghi log lỗi và kết thúc yêu cầu hiện tại. |


```
@startuml

box "Application" #LightGreen
    participant "Application Service" as APP
end box

box "Log Ingestion & Processing" #LightBlue
    participant "Filebeat" as FB
    participant "Logstash" as LS
    database "Elasticsearch" as ES
end box

box "Backend" #LightYellow
    participant "Collector Backend" as BE
end box

box "Client" #LightGreen
    participant "UI / Frontend" as UI
end box

== Bước 1: Sinh log ==

APP -> APP: Ghi log\nvào file

== Bước 2: Thu thập & Chuẩn hóa Log ==

FB -> FB: Đọc file log và gộp \nstacktrace nhiều dòng
FB -> LS: Chuyển tiếp log qua \nBeats Protocol (Cổng 5044)

activate LS

LS -> LS: Áp dụng Grok Filter để bóc tách trường\n(timestamp, level, trace_id, message...)
LS -> LS: Chuẩn hóa dữ liệu log

== Bước 3: Lưu trữ & Chuyển tiếp ==

par Luồng lưu trữ và luồng phân phối hoạt động độc lập

    LS -> ES: Lưu log vào Elasticsearch

else Luồng gửi Backend

    LS -> BE: HTTP POST log về backend

    activate BE

    BE --> LS: 202 Accepted

    deactivate LS

end

== Bước 4: Xử lý bất đồng bộ & Phát tán Real-time ==

activate BE

note over BE
Xử lý bất đồng bộ:
- Kiểm tra dữ liệu log
- Chuẩn hóa thông tin
- Xác định kênh phân phối
end note

BE -> UI: Broadcast realtime

deactivate BE

@enduml
```
